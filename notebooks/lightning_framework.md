# The Training Framework: Hydra Configs, Lightning, and How to Run Things

This document is about wiring of the models and not about the model or the losses themselves. `heinzlenet.md` explains why the network looks the way it does. `training.md` explains why the loss terms exist and how they're scheduled.
This document explains how `scripts/train_mich.py` turns a pile of YAML files into a running PyTorch
Lightning training job, how to launch one, what to watch while it runs, and which config file to edit for
a given change (the part you'll come back to most often).

---

## 1. The big picture: one root config, composed from groups

Every training run starts from a single root file, `config/mainconfig.yaml`, resolved by
[Hydra](https://hydra.cc/). Hydra's job is to merge a set of YAML "config groups" into one tree before your
Python code ever runs. `mainconfig.yaml`'s `defaults` list says which file is pulled in for each group, and
the list's own comment states the rule for conflicts: **"Order indicates overwriting."** Later entries in
the list win if two groups set the same key.

```yaml
# config/mainconfig.yaml
defaults:
  - _self_
  - private: private.yaml
  - paths: default.yaml
  - trainer: default.yaml
  - model: default.yaml
  - datamodule: default.yaml
  - loggers: default.yaml
  - hydra: default.yaml
  - callbacks: default.yaml
  - experiments: null       # opt-in override bundle, see §5
```

Each group is a directory under `config/`, and each entry names one file inside it (without the `.yaml`).
`experiments: null` means no experiment preset is applied by default. You only get one by passing
`experiment=<name>` on the CLI (§5).

| Group | File | Controls |
|---|---|---|
| `private` | `config/private/private.yaml` (gitignored, see README "Setup") | Your local paths and W&B entity |
| `paths` | `config/paths/default.yaml` | Derives `${paths.full_path}`, the one output directory everything else writes into |
| `trainer` | `config/trainer/default.yaml` | Raw `pytorch_lightning.Trainer` kwargs: epochs, devices, precision, gradient clipping |
| `model` | `config/model/{default,longfreq,supervised}.yaml` | Full network architecture and loss config, see `02_heinzlenet_model.ipynb` |
| `datamodule` | `config/datamodule/default.yaml` | Which HDF5 file to train on, batch size, train/val/test split |
| `loggers` | `config/loggers/default.yaml` | Weights & Biases logger settings |
| `callbacks` | `config/callbacks/default.yaml` | Checkpointing, LR monitor, model summary |
| `hydra` | `config/hydra/default.yaml` | Hydra's own behaviour: working-directory change, run-dir location, log formatting |
| `experiments` | `config/experiments/{debug_offline,debug_online,default}.yaml` | Optional bundle of overrides across all the above, applied as one preset |

Every value in the composed tree can be overridden from the command line with dot syntax. This is the
mechanism behind every example below.

## 2. What each group actually contains, and what you'll edit

**`config/private/private.yaml`**: not tracked by git. Created once per machine by copying
`config/private/default.yaml` and filling in `entity` (your W&B username/team), `output_dir` (where
checkpoints/logs go), `data_dir` (where simulation HDF5 files live), `root_dir` (repo root). See the
README's "Setup → Private config" section for the exact steps. This file is the only place absolute,
machine-specific paths belong.

**`config/paths/default.yaml`**: pure interpolation glue, nothing to edit directly:
```yaml
full_path: ${private.output_dir}/${project_name}/${network_name}/
```
`project_name` and `network_name` (set at the top of `mainconfig.yaml`, or overridden per run) actually
name a run's output directory. See §6.

**`config/trainer/default.yaml`**: standard Lightning `Trainer` kwargs. Edit this (or override on the
CLI) for `max_epochs`/`min_epochs`, `devices`/`accelerator`/`strategy` (multi-GPU), `precision`
(`bf16-mixed` by default), `gradient_clip_val`, `check_val_every_n_epoch`.

**`config/model/*.yaml`**: architecture dimensions, the loss config (all the `lambda_*`/warmup/delay
keys described in `training.md`), the optimizer/scheduler, and `learnable_physio`. Fully covered in
`02_heinzlenet_model.ipynb`. Three variants exist: `default.yaml` (3-layer, full drainage), `longfreq.yaml`
(single-layer, no drainage, used by both `experiments/debug_*.yaml` presets), and `supervised.yaml` (the
non-physics baseline, `SupervisedMICH`).

**`config/datamodule/default.yaml`**: which HDF5 file to read (`data.path`, generated by `run_sim.py` as
in `01_running_simulations.ipynb`), the train/val/test split (`split.*`, either fraction- or count-based),
and `DataLoader` settings (`loader.batch_size`, `loader.num_workers`, and so on). This is the file you edit
to point a training run at a different simulated scenario.

**`config/callbacks/default.yaml`**: two `ModelCheckpoint`s (`last`, saved every 100 steps; `best`,
tracking `val/loss/total`), an `LearningRateMonitor`, and a `ModelSummary`. Rarely needs editing beyond the
monitored metric or checkpoint frequency.

**`config/loggers/default.yaml`**: a single `WandbLogger`. `offline: True/False` is the main thing you'd
flip by hand if not using an experiment preset (§5 does this for you).

**`config/hydra/default.yaml`**: sets `job.chdir: True` (the process `cd`s into `${paths.full_path}` at
startup, so any relative path written after this point is relative to that directory, not the repo root)
and `run.dir: ${paths.full_path}` (Hydra's own per-run bookkeeping lands in the same directory as
everything else, rather than its usual default `outputs/<date>/<time>/`).

## 3. Running a default training

```bash
python scripts/train_mich.py
```

This uses every `default.yaml` in the table above, with no `experiment=` preset. Override anything with
dot syntax; there's no need to edit files for a one-off change:

```bash
# change architecture width and epoch budget for one run
python scripts/train_mich.py model.heinzle_net.layer_mixing_config.C=32 trainer.max_epochs=100

# point at a different simulated scenario, with a bigger batch size
python scripts/train_mich.py \
    datamodule.data.path=${paths.data_dir}/simulations-three_layer_multi_source_multi_active.h5 \
    datamodule.loader.batch_size=64

# give the run a name that shows up in the output path and in W&B
python scripts/train_mich.py project_name=mich-bold-inversion network_name=my_first_run
```

`project_name`/`network_name` matter more than they look: they determine both the W&B project/run-group
name and the output directory (`${paths.full_path}`, §6). Always set `network_name` to something
descriptive per run, since checkpoints for two runs with the same `project_name`/`network_name` land in the
same directory and overwrite each other's `last.ckpt`.

## 4. What actually happens inside `train_mich.py`

Reading this once is worth it, because several steps are easy to miss if you only skim the config files.

1. **Resume check** (`cfg.resume.state`). See §7; skip on a first read.
2. **Instantiate the datamodule** (`hydra.utils.instantiate(cfg.datamodule)`). This opens the HDF5 file at
   `datamodule.data.path` and computes the train/val/test split.
3. **`_inject_sim_physics`**: reads `datamodule.sim_config` (the JSON blob every simulation HDF5 carries in
   its `/meta` group, written by `run_sim.py`) and overwrites `cfg.model.haemo`, `cfg.model.acquisition`,
   `cfg.model.V0`, `cfg.model.psf_fwhm`, `cfg.model.loss_config.order`, `cfg.model.L`,
   `cfg.model.out_channels`, and the decoder's `signals` list. This is why you never hand-edit
   physiological constants in `config/model/*.yaml` when switching datasets: they're always derived
   fresh from whatever file `datamodule.data.path` points to, so the model's assumed physics can never
   silently drift out of sync with the data it's being trained on.
4. **`_inject_scheduler_steps`**: auto-computes `cfg.model.scheduler.T_max` (the cosine LR schedule's
   period) from the actual train-split size, batch size, and `trainer.max_epochs`, unless you've pinned
   `model.scheduler.T_max` explicitly on the CLI.
5. **Instantiate the model** (`hydra.utils.instantiate(cfg.model)`). Builds `MICH` (or `SupervisedMICH`)
   with the now-physics-corrected config, exactly as in `02_heinzlenet_model.ipynb` §5, just automated.
6. **Instantiate callbacks, loggers, then the trainer.**
7. **`log_hyperparameters`**: logs the fully-resolved config plus parameter counts to W&B.
8. **`save_config`**: writes the fully-resolved config (every `${...}` interpolation already substituted,
   unlike Hydra's own snapshot) to `<full_path>/full_config.yaml`. This file is what `eval_mich.py` later
   depends on (§8): it's the only complete record of what a checkpoint was actually trained with.
9. **`trainer.fit(model, datamodule, ckpt_path=cfg.ckpt_path)`**, unless `cfg.train=False`.

## 5. Debug presets: `experiment=debug_offline` / `experiment=debug_online`

`config/experiments/{debug_offline,debug_online}.yaml` are override bundles. Each swaps in the smaller
`longfreq` model variant and re-overrides trainer/model/datamodule/logger keys in one shot, so you don't
have to remember a dozen individual overrides when you just want to sanity-check that something runs.

```bash
# fastest possible sanity check: 8 train / 2 val samples, 3 epochs, no physics loss, wandb offline
python scripts/train_mich.py experiment=debug_offline

# a step up: physics loss active with a short warmup, wandb synced online, 32 train samples
python scripts/train_mich.py experiment=debug_online trainer.devices=2
```

`debug_offline` is the one to reach for after any architecture/loss code change, before committing to a
full run. It should complete a few epochs in well under a minute. `debug_online` is a step closer to a
real run (it keeps the physics loss active with a short warmup) while still being small enough to iterate
on quickly, and it syncs to W&B so you can check the dashboards actually populate correctly.

You can still override on top of a preset. Hydra applies explicit CLI overrides after the whole defaults
list is composed, so any `key=value` you add after `experiment=debug_offline` wins over that preset's own
values, as in the `trainer.devices=2` example above.

## 6. Where a run's outputs actually land

Everything from one run lives together under:
```
${private.output_dir}/${project_name}/${network_name}/
```
(that's `${paths.full_path}`). On disk, a finished run directory looks like:

```
<full_path>/
├── .hydra/
│   ├── config.yaml       # Hydra's own snapshot, unresolved interpolations
│   ├── hydra.yaml
│   └── overrides.yaml    # exactly which CLI overrides were passed
├── full_config.yaml      # the fully-resolved config; what eval_mich.py reads (§8)
├── train_mich.log        # this run's log (overwritten, not appended, on rerun)
├── checkpoints/
│   ├── last.ckpt
│   └── best.ckpt
└── wandb/                # local W&B run cache
```

If you also run `scripts/eval_mich.py` against this run, its output (plots, GIFs, metrics) lands in a
`results/` subdirectory alongside the above by default (`--local-out`, §8). A training run's directory and
its evaluation results are colocated, not separate top-level concepts.

A genuine Hydra multirun sweep (`-m`/`--multirun`, §9) is a different, separate output convention.
`config/hydra/default.yaml` only repoints `run.dir` (single-run), not `sweep.dir`, so multirun jobs land in
Hydra's own default `multirun/<date>/<time>/<job_index>/` at the repo root instead.

## 7. Resuming a run

There are two distinct mechanisms, and it matters which one you use:

- **`ckpt_path=<path>`**: the plain Lightning mechanism. Loads model + optimizer state from a checkpoint,
  but you're responsible for re-specifying every other config value yourself (architecture, data path, and
  so on). If anything drifts from what the checkpoint was trained with, you get a silent mismatch.
- **`resume.state=True`** (with `resume.ckpt_flag`, default `"*last*"`): reconstructs the entire
  original config from that run's own `full_config.yaml` (see `reload_original_config` in
  `src/mich/utils/hydra_utils.py`), then re-applies only whatever CLI overrides you pass this time, and
  auto-picks the matching checkpoint file. This is the mechanism to reach for. It guarantees the resumed
  run's architecture and physics constants exactly match the checkpoint, because they're read back from
  the same file the original run wrote.

```bash
# resume the exact run at project_name=mich-bold-inversion, network_name=my_first_run,
# picking up from its last checkpoint, for 200 more epochs
python scripts/train_mich.py project_name=mich-bold-inversion network_name=my_first_run \
    resume.state=True trainer.max_epochs=200
```

## 8. Evaluating a trained checkpoint

`scripts/eval_mich.py` is a plain `argparse` script, not a Hydra entry point. It reads
`<run_dir>/full_config.yaml` (written by `train_mich.py`, §4 step 8) to reconstruct the model, so any
checkpoint not produced by `train_mich.py` (or missing that file) isn't supported.

```bash
# resolve a run by project/network name, evaluate on its own held-out test split
python scripts/eval_mich.py --project mich-bold-inversion --network single_layer_single_source

# point at a checkpoint directly, evaluate on a specific file or directory of files
python scripts/eval_mich.py -c results/.../checkpoints/best.ckpt -d data/gridsearch/

# compare a PINN run against a supervised baseline on the same data
python scripts/eval_mich.py --project P --network pinn_run \
    --compare-checkpoint results/P/supervised_run/checkpoints/best.ckpt -d data/a.h5

# from a saved flat-YAML config (config/eval/*.yaml); CLI flags still override individual keys
python scripts/eval_mich.py --config config/eval/single_layer_single_source.yaml -n 16
```

`config/eval/*.yaml` files are flat YAML whose keys map 1:1 onto the script's CLI flags (for example
`ckpt_tag: best`, `data: [data/simulations-single_layer_single_source.h5]`, `n_samples: 8`). This is a
convenient way to save a commonly-repeated eval invocation without retyping every flag.

## 9. Sweeping: Hydra multirun

Hydra's built-in multirun (`-m`/`--multirun`) runs the same entry point once per combination of a
comma-separated override. This is useful for hyperparameter sweeps. It isn't currently exercised anywhere
in `scripts/train_mich.py`'s own docs. The `multirun/` directory at the repo root, gitignored, is evidence
it has been used, likely for `run_sim.py` parameter sweeps as in `01_running_simulations.ipynb` §4. The
same `-m` flag works for `train_mich.py` too:

```bash
python scripts/train_mich.py -m network_name=sweep_c16,sweep_c32,sweep_c64 \
    model.heinzle_net.layer_mixing_config.C=16,32,64
```

Treat this as standard Hydra behaviour (see the [Hydra multirun docs](https://hydra.cc/docs/tutorials/basic/running_your_app/multi-run/))
rather than something specific to this repo. Nothing here changes how it works, beyond `network_name`
needing to vary per job (otherwise every job's output would collide in the same `${paths.full_path}`).

## 10. What to watch while a run trains

Everything is logged to Weights & Biases (`loggers.wandb`, project = `project_name`, run name =
`network_name`). At minimum, watch:

- **`val/loss/total`**: the metric `best_model_checkpoint` tracks. This is what "the run is improving"
  means in the most basic sense.
- **The individual loss terms** (`train/loss/data`, `train/loss/physics`, and so on): particularly whether
  the physics loss actually engages after its `warmup_steps_physics`/`delay_steps_physics` (see
  `training.md` §7), and whether the total loss doesn't suddenly collapse toward the trivial resting-state
  solution described in `training.md` §4 right as the physics loss ramps up.
- **The validation image panels**: logged once per `image_log_every_n_val_calls` validation epochs
  (`config/model/*.yaml`), comparing predicted vs. ground-truth BOLD/neural activity and the full set of
  latent states at the source voxel (`training.md` §8). This is the primary qualitative signal. Loss curves
  alone can look fine while the recovered latents are visually nonsensical.
- **GPU utilization / step time**, if you're debugging throughput rather than convergence. The
  `debug_online` preset (§5) is the fast way to check a config change didn't regress speed before
  committing to a multi-hour run.

## 11. Cookbook: "I want to change X"

| I want to... | Edit / override |
|---|---|
| Train on a different simulated scenario | `datamodule.data.path=<file>.h5` |
| Change how many epochs / GPUs | `trainer.max_epochs=`, `trainer.devices=` |
| Make the network bigger/smaller | `model.C`, `model.c_enc`, `model.c_dec` (see `02_heinzlenet_model.ipynb` §2) |
| Change a loss weight or its warmup | `model.loss_config.lambda_*` / `warmup_steps_*` / `delay_steps_*` (see `training.md`) |
| Make a physiological constant learnable | `model.learnable_physio.<name>=true` |
| Quickly sanity-check a code change | `experiment=debug_offline` |
| Name a run so it doesn't collide with others | `project_name=`, `network_name=` |
| Resume a specific run | `project_name=... network_name=... resume.state=True` |
| Sync to W&B or not | `experiment=debug_online` / `debug_offline`, or `loggers.wandb.offline=` directly |
| Change where checkpoints/logs go | `config/private/private.yaml`'s `output_dir` (once, per machine) |
| Evaluate a trained checkpoint | `scripts/eval_mich.py --project ... --network ...` (§8) |
